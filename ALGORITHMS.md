# Описание эндпоинтов и алгоритмов сервиса рекомендаций

Документ описывает логику работы каждого API эндпоинта рекомендательного сервиса (rec-service).

## 1. GET /recommendations/{user_id}?n={count}
**Функционал:** Возвращает персональные рекомендации фильмов для заданного пользователя.

**Алгоритм (Item-based Collaborative Filtering):**
1. **Загрузка истории пользователя:** Сервис запрашивает из базы данных все оценки, которые поставил данный пользователь.
2. **Проверка холодного старта (cold start):** Если у пользователя слишком мало оценок (в текущей конфигурации — меньше 5), алгоритму недостаточно данных для формирования точных рекомендаций. В этом случае происходит автоматическое переключение на алгоритм выдачи популярных фильмов (та же логика, что описана для эндпоинта `/popular` ниже), и клиенту возвращается результат его работы.
3. **Отбор кандидатов:** Если оценок достаточно, для каждого оцененного пользователем фильма из оперативной памяти извлекаются его $K$ (в текущей конфигурации $K=30$) ближайших соседей (наиболее похожих фильмов). Если пользователь оценил 20 фильмов, алгоритм анализирует до 600 кандидатов. Эти "соседи" были заранее вычислены скриптом инициализации базы данных на основе косинусной меры сходства.
4. **Фильтрация:** Из списка кандидатов удаляются фильмы, которые пользователь уже оценил.
5. **Расчет ожидаемой оценки:** Для каждого фильма-кандидата алгоритм вычисляет итоговый балл по формуле средневзвешенного значения:
   - Алгоритм находит, с какими из уже оцененных пользователем фильмов этот кандидат имеет математическое сходство.
   - Оценка, которую пользователь поставил известному фильму, умножается на коэффициент косинусного сходства между известным фильмом и кандидатом. Косинусное сходство (от 0 до 1) определяет "вес" этой оценки: чем сильнее похожи векторы оценок двух фильмов у всех пользователей системы, тем выше вес.
   - Полученные взвешенные оценки суммируются и делятся на общую сумму всех задействованных коэффициентов сходства (нормализация, чтобы вернуть итоговый балл в рамки от 1 до 5).
   - В результате алгоритм математически отдает предпочтение тем фильмам-кандидатам, которые максимально похожи на высоко оцененные фильмы пользователя, и пессимизирует те, что похожи на низко оцененные.
6. **Выдача:** Список кандидатов сортируется по убыванию ожидаемого балла, и пользователю возвращаются лучшие $N$ результатов.

## 2. GET /recommendations/{user_id}/popular?n={count}
**Функционал:** Возвращает список самых популярных фильмов, которые пользователь еще не смотрел.

**Алгоритм (Популярные фильмы):**
1. Сервис запрашивает из БД список идентификаторов фильмов, которые уже просмотрены (оценены) данным пользователем.
2. Делается SQL-запрос к таблице фильмов (с уже предрасчитанными в базе `ratings_count` и `avg_rating`). Чтобы получить достаточное количество непросмотренных фильмов, сервис запрашивает топ-$M$ популярных фильмов, где размер выборки $LIMIT = N + \{\text{количество уже просмотренных фильмов}\}$.
3. Исключение просмотренных происходит на уровне кода сервиса (после SELECT-запроса): сервис итерируется по возвращенным из БД фильмам, отсекает уже просмотренные, и собирает список до тех пор, пока не наберет ровно $N$ результатов.
4. Выбираются фильмы с наибольшим параметром популярности. В сервисе популярность рассчитывается как `avg_rating * ln(ratings_count + 1)`, что позволяет балансировать между высоким рейтингом и количеством оценок. В JSON-ответ записывается чистый `avg_rating`.
5. Возвращаются $N$ лучших результатов. Это базовая стратегия (fallback), которая хорошо работает для новых пользователей.

## 3. GET /similar/{movie_id}?n={count}
**Функционал:** Возвращает фильмы, которые алгоритмически наиболее похожи на запрошенный фильм.

**Алгоритм (Косинусное сходство по матрице оценок):**
1. Сервис обращается к загруженной в оперативную память модели сходства (Similarity Model).
2. Находит заданный `movie_id` и достает $N$ его ближайших соседей.
3. Показатели сходства были заранее рассчитаны на этапе инициализации БД (`compute_similarity.py`):
   - Строится разреженная матрица `пользователь-фильм` с помощью `scipy`.
   - Вычисляется косинусное сходство (`cosine similarity` из `scikit-learn`) между столбцами (фильмами).
   - Ближайшие соседи записываются в БД и кэшируются сервисом при старте.
4. Для найденных ID соседей из БД подтягиваются их текстовые названия, и ответ отдается клиенту. Если фильма нет в матрице, возвращается ошибка 404.

## 4. GET /health
**Функционал:** Проверка состояния приложения (Liveness / Readiness Probe).

**Алгоритм:**
1. Обрабатывает простой HTTP-запрос.
2. Проверяет, была ли успешно загружена в память модель сходства (матрица соседей).
3. Возвращает JSON со статусом сервера (например, `status: "ok"`) и переменной состояния модели (`model_loaded: true`). Используется Docker Compose и оркестраторами для проверки "здоровья" контейнера.

## 5. GET /metrics
**Функционал:** Предоставляет внутренние метрики работы приложения для мониторинга в формате Prometheus.

**Алгоритм:**
1. При обращении к эндпоинту встроенные инструменты метрик (клиент Prometheus) собирают текущее состояние среды.
2. Выводятся данные о потреблении памяти, процессора, количестве обработанных HTTP-запросов, распределении времени ответов (гистограммы latency) и состоянии пула соединений с БД.
3. Форматирует данные в текстовом виде (Plain Text), который сервер Prometheus регулярно забирает (pull) для отображения на дашбордах.
