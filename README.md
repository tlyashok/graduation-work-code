# Рекомендательный сервис: Python и Go

Здесь представлен весь исходный код к выпускной работе «Сравнительный анализ и оптимизация
производительности RESTful API рекомендательного сервиса на примере Python и Go». Система состоит из следующих компонентов: веб-приложение с интерфейсом, рекомендательный микросервис в двух реализациях — на Python и на Go — и обвязка для нагрузочного
тестирования, на которой получены все числа из третьей главы.

Тестирование проводилось таким образом: брался один и тот же рекомендательный сервис, написанный на
Python и на Go, по одному и тому же алгоритму, — и в одинаковых условиях замерялось, сколько
запросов в секунду выдерживает каждая реализация. Затем обе реализации профилировались, и каждая следующая итерация устраняла узкие места, найденные на предыдущей.

## Что где лежит

**shared** — общие части, которые не зависят от языка и не меняются от итерации к итерации.
Внутри `web-app` — веб-система на FastAPI и Jinja2 (каталог фильмов, страницы с описанием,
выдача рекомендаций); через неё пользователь и обращается к сервису. В `db` собрано всё про
данные: скрипт инициализации семи таблиц `init.sql`, скрипт `seed_movielens.py`, наполняющий базу данных набором MovieLens 25M, и скрипт офлайн-расчёта модели похожести фильмов
`compute_similarity.py`.

**services** — сам рекомендательный микросервис, разложенный по трём итерациям оптимизации.
В каждой итерации лежат две независимые реализации, `python` и `go`. Алгоритм рекомендаций
в них одинаков — отличаются только язык, драйвер базы и то, что оптимизируется на данном шаге.

**deploy** — конфигурации запуска через Docker Compose, по одной на каждую итерацию. В каждой
есть `Makefile` с короткими командами, шаблон файла переменных окружений `.env.example` и папка `compose` с описанием контейнеров (сервис, база, nginx, Prometheus, а для второй итерации — ещё и Redis).

**testing** — нагрузочное тестирование и всё, что с ним связано: сценарии для k6, управляющие
скрипты прогонов и профилирования, скрипты анализа и построения графиков, сохранённые
результаты, диаграммы пламени, готовые графики и собранный отчёт. Здесь же, в `cloud`, лежат
скрипты подготовки арендованного сервера, на котором сделан финальный замер, и запуска на нём полного прогона.

Весь эксперимент выполняется одним скриптом `run_final.sh`: он четыре раза повторяет полный
замер ёмкости, снимает профили процессорного времени и собирает итоговый отчёт. Остальные
управляющие скрипты вызываются из него, запускать их по отдельности не требуется. Из папки
`analysis` в прогоне участвует только сборщик отчёта `build_report.py`; остальные скрипты
обосновывают методику и оформляют её результаты: `session_weights.py` считает доли обращений
к каждому эндпоинту, `movielens_stats.py` считает распределение активности пользователей
в наборе данных, а `build_capacity_chart.py` строит итоговую диаграмму ёмкости для текста работы.

## Как запустить

Требования: Docker, Python 3.12, Go 1.24 и make.

1. Выбрать итерацию и зайти в её папку, например `deploy/1-async-pooling`.
2. Скопировать `.env.example` в `.env` и при необходимости поправить значения.
3. Один раз загрузить данные и построить модель похожести: `make init`.
4. Поднять систему с нужной реализацией: `make python` или `make go`.

После этого веб-система открывается в браузере по адресу http://localhost. Остановить всё —
`make down`, полный сброс с удалением данных — `make clean`.

## Три итерации оптимизации

**Итерация 0** — отправная точка: синхронный драйвер базы, пул из 5 подключений к базе данных. Намеренно неоптимизированная конфигурация, на которой видно исходное узкое место.

**Итерация 1** — устраняем главные тормоза, общие для обоих языков: добавляем индексы (по
идентификатору пользователя и по популярности фильма), увеличиваем пул до 10 соединений, а в
Python переходим с синхронного драйвера на асинхронный (asyncpg). После этого база перестаёт
быть узким местом, и на первый план выходит сам язык.

**Итерация 2** — добавляем кэш в Redis по схеме отложенной загрузки (cache-aside): часто
запрашиваемые рекомендации возвращаются из кэша, минуя базу и пересчёт. Рассмотрены два
времени жизни записи — 60 и 600 секунд.

## Результаты

Все числа из работы получены на арендованном облачном стенде; каждая конфигурация
прогонялась четыре раза, в работе приводятся медианы по этим прогонам. Результаты лежат
в `testing/results`, а собранный наглядный отчёт со всеми таблицами, графиками и профилями —
в `testing/results/report/index.html` (открывается в браузере).
